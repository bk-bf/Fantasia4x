A map of the codebase · read to understand, not to build

The Fantasia4x Codebase Atlas

A turn-based colony sim: raise a race of pawns, work the land, craft and build, weather a living world. Underneath, ~2,500 functions across 328 files, run 60 times a second inside a background worker.

This is an explanation, not a tour. It starts where the codebase is densest — the handful of pieces everything else leans on — and works outward, so the shape comes before the detail. Every claim is anchored to a file or a codeName, and the depth is always one click away.

The frame · five floors

Everything hangs off five layers, and things only fall downward

Above the load-bearing beams sits a strict stack. Each floor may call the ones below it and never the ones above. Between the top two floors is a real wall: the game runs in a background thread, so that hop is a serialized message, not a function call.

The beams · cross-cutting infrastructure

The load-bearing pieces, up close

The pieces from the map that aren't any one subsystem — they hold up all of them.

Seeded randomness

85 callerscore/rng · SeededRng.random

Every dice-roll in the game — a stat gain, a loot drop, where a mob wanders — draws from one seeded generator. Same seed, same sequence, forever.

the real story

This is the single most-called function in the codebase, and it's why the whole sim is reproducible. The headless test harness, the AUDIT sweeps, and save-replay all rely on it: a scenario plus a seed plus a command script produces a byte-identical world every time. That guarantee only holds because nothing anywhere calls Math.random() — and because ids are turn-stamped (t42-…) rather than time-stamped. One stray un-seeded draw silently breaks every replay.

The lookups

73 · 54 · 29ItemService · BuildingService · core/distance

Items, buildings and tiles are referenced everywhere by id or coordinate. So the id-to-thing lookups (getItemById, getBuildingById) and the grid-distance helper (manhattan) end up near the top of the call graph — plumbing that every feature threads through.

the real story

These being hubs is healthy — a shared lookup is exactly what you want. It's why ids-not-objects is the pervasive style: the world stores 'oak_log', not a copy of the oak definition, and resolves it on demand. The cost is that these functions are on nearly every hot path, so they're written to be cheap dictionary reads. The one smell nearby: manhattan and a few friends (clamp, dist) are defined in two places each — see the graph's own findings below.

The gameState store

69 linksstores/gameState

The one picture the whole UI reads from. The worker hands it a fresh snapshot on every flush; the screens subscribe and redraw. It never computes anything itself.

the real story

It's also a 75-function god-module: subscribe/command plumbing, ~40 game actions (addItem, haulTileToStockpile, saveGame, loadScenarioPreset), and a wall of derived selectors all live in one file. The read side is deliberately dumb — it holds the latest worker snapshot and re-emits on flush — but the action side (which turns a click into a command) is the natural other half of the write path below. Persistence rides here too, now via IndexedDB (saveManager), not localStorage as the old notes claim.

The gate — GameStateManager

the only writercore/GameState

Every change to the world funnels through one gatekeeper. Nothing writes to the world directly; it all goes through methods like addResource, updatePawn, addBuilding.

the real story

Having one mutation surface is what keeps a 2,500-function sim reasoning-about-able: there is always a single, known place where reality changes. The one sanctioned exception is the hot per-tick path, where a few phases mutate entity fields in place for speed (the immutable spread/`.map()` style was measured at ~12.5× the tick cost). That exception is deliberate and documented — but it's also the seam where the “growth mutates in place while acting copies” mismatch lives (see the Growth subsystem).

The turn-taker — GameEngineImpl

43 linkssystems/GameEngineImpl.processGameTurn

One tick is one call to processGameTurn. It runs 19 fixed steps in order — needs, upkeep, jobs, buildings, pawns, growth, the world, entities, combat, events — then hands the picture over. It holds no rules; it sequences the specialists.

the 19 steps, and why the order is load-bearing

weather → hunger/thirst/sleep → food spoils → gear wears → wood dries → research → reserve craft parts → rebuild the to-do list → buildings & fires → furnaces cook → pawns act → pawns grow → plants regrow → crops grow → wild spreads → animals & monsters → combat → clear the dead → world events. Each step reads what the one before it produced, so the order can't be shuffled. (The old notes say 13 steps; the code runs 19 — it grew.) Steps 11 and 12 being separate is the thread the next section pulls on.

The write path — commands.ts

104 fns · 88 actionssim/commands

You never edit the world; you send it a named request. Every possible action — chop, equip, draft, research, haul — is one entry in a list of 88. The request is plain data, so it crosses the worker wall safely and replays identically.

the real story

One registry, three users: the buttons you click, the /api/sim HTTP driver, and the invariant test suite all steer through the same list. A command is {type, payload?, save?} with no closures, so structured-clone survives it. It's a 104-function god-module by count — but a flat registry is meant to be flat, so this one earns its size. The value it buys is enormous: the entire AUDIT is driven by scripting these commands headlessly.

The answer machines — stats & bonuses

30 · —PawnStatService.evaluateStat · systems/ModifierSystem

Two “one right way” rules. evaluateStat is the single place a pawn's stat is computed; ModifierSystem is the single place gear/building/trait bonuses are tallied — and it always shows its receipts (sources[]).

the real story

Work skill has its own single home too — pawnStatService.getWorkModifiers (speed/yield/quality from stats.jsonc × body capacities × traits). An older second work-calc path inside ModifierSystem was deleted on purpose; don't reintroduce it. These “exactly one place” rules are the reason a new feature is usually new data, not a new code path — the calculation already exists, you just feed it.

The domains · one expert each

The subsystems

Each sits above the beams. Summary first; expand for the depth; ◆ marks a real design tension found in the code.

Pawns · the state machine

systems/PawnStateMachine

Each pawn is always in exactly one of 22 states — Idle, Working, Hauling, Eating, Sleeping, Fighting, Fleeing, Hunting, Collapsed, Panicking… A lookup table picks the handler, so adding a state is one line. Health and bleeding are checked first every tick, even for drafted pawns.

◆ why is it like this?The file's own header says the pawn AI “was decomposed out” as a 2026 hotspot — yet 1,851 lines remain. The easy 15 handlers left; the hard half (bleed, heal, death, collapse, aura) never did. “State-machine tick” is really state machine + a full medical simulation.

Pawns · growth, aging & lineage

PawnGrowthService · core/Lineages

Once per in-game day, each pawn banks a growth offer (four a year: three seasonal + a birthday that also ages them). An offer rolls all six stats; you later pick two to keep. It's opt-in meta-progression, not something a pawn does while acting — which is why it's a separate step from “pawns act,” not an FSM state.

why steps 11 and 12 are separate — the deeper answer

processDay isn't really “growth.” It's the catch-all “once a day” hook: in one loop it (a) banks stat offers, (b) ages the pawn, and (c) drives the entire lineage / awakening system. So the separate phase isn't about stat growth at all — it's wherever anything day-granular got dumped. Add that it mutates pawns in place while step 11 threads immutable copies, and the double day-gating (the engine loops days while processDay re-derives the date itself), and the phase reads as grown-not-designed.

◆ and a second oneLineage keeps its own duplicate index of the trait catalog purely to dodge a circular import between Lineages and Culture — a layering problem papered over with two copies of the same data.

Health, wounds & death

core/BodyParts · wounds.jsonc

A real body: parts nest and contain each other, destroying a vital one is instant death, and severing cascades to everything inside. Wounds stack by damage type; healing is deliberately slow, sped by sleep, food and tending. Three ways to die: lethal anatomy, bleeding out, or a downed Collapsed state.

◆ why is it like this?The death rule lethalAnatomyCause is called from three places — combat's injury step, and the per-tick pawn and mob reapers independently. “Dead means dead” holds only if all three stay in sync. Correctness-by-repetition, not a single choke point.

Combat

systems/Combat (2,746 lines)

Weapons pull a damage profile; armour is layered (twelve slots map to body parts, mitigation walks outer-to-inner); shields are a real second defence — block, parry, bash — distinct from dodge. A two-hander that also straps a shield is penalised, not forbidden.

◆ why is it like this?Combat isn't pure damage resolution — _applyInjuryToEntity reaches in to credit lineage awakening progress mid-crush-hit, and the file also owns durability, natural weapons and the death rule. And because block is CON + body-mass, every mob now negates ~1.8% of melee with no shield — unintended “bracing.”

Equipment & carrying

ItemService.getCarryBudget

Carry budget is a strength-scaled slice of the pawn's own body weight; packs and carts add capacity, and even armour adds a little volume. Volume, oddly, ignores strength entirely while weight scales on it.

◆ why is it like this?getCarryBudget(pawn, _state) takes a GameState it never uses — and one caller fabricates an empty {} as GameState just to satisfy the signature. The function advertises a dependency that doesn't exist, forcing a caller to lie.

Production · reserve-and-fetch

services/jobs/craft · ADR-016

Items are always physical. A craft order starts as a wish; each tick it tries to reserve all its inputs at once, committing only if every one is free. Once staged on the station tile, a pawn fetches, walks and works it. Furnaces are passive — no pawn, they cook on their own if lit.

◆ why is it like this?Whether a craft needs a pawn is decided in two independent places that must agree — the job generator skips passive stations, and the engine re-derives the same fact. Miss one and a furnace either produces nothing or spawns a phantom job.

Crafting · stations, tiers & recipes

recipeService · recipes.jsonc

~330 recipes at stations that ladder up in tier; a lower tier stays usable after a higher one is built. Butchery stations give a yield bonus (more off a carcass); tool and cooking stations give speed instead.

◆ from the audit“How is X made?” returns the first recipe producing it — so 25 recipes sharing an output were unreachable (18 fixed, 8 still shadowed). And the steel recipes are metallurgically inverted: a finery forge decarburises iron in reality, it never made steel.

Materials · the rulebook of stuff

core/materialProperties · items.jsonc

Materials are a fixed, hand-authored table keyed by item id (oak vs pine, granite vs marble). The leather rework split one-leather-per-animal into 17 per-animal leathers — buckskin, oxhide, bearhide, direwolf — each with its own durability, warmth and beauty.

◆ why is it like this?“Procedural metals” isn't endgame-open — it's not modelled at all; materials are a static lookup, and the property cache assumes static-forever. This is a running theme: almost nothing was born general (see how it grew).

Jobs & work

database/pawns/jobs.jsonc · ADR-017

Eleven job types (harvest, haul, fetch, construct, craft, refuel, plant…) are declared as data, each bound to a generate/complete handler by id. Work category, priority, UI and gating all flow from the definition. You can also right-click a pawn to force a job.

the real story

Colony jobs (a JobDef each) and the FSM's internal kinds (eat, sleep, need) are two separate registries — the FSM kinds have no JobDef. The old duplicated “job type → work category” switch was deleted; everything now routes through getJobWorkCategory. A “make X” is almost always just a new recipe, not a new job type — genuinely new verbs (like fetch once was) are rare and guarded by a registry test.

The social layer

SocialService · core/Social

Pawns form culture-seeded relationships that drift through stages, hold conversations, fall in love, and feed a mood layer. Big jumps arrive via hooks — a death, a rescue, a battle fought together. The heavy pass runs once per in-game day, in the events phase.

◆ why is it like this?The daily pass is O(n²) over living pawns (~1,225 pairs at 50 pawns). There's no spatial partition — the design leans on a hard pawn-count ceiling, and the once-a-day gate is the mitigation.

Kingdoms, visitors & trade

KingdomService · core/Kingdom

Other kingdoms drift their own traits, make hidden-knowledge contact, and send visitors and caravans you can barter with. It all ticks once per in-game day, beside the social pass.

◆ why is it like this?A single arrival clock paces both visitors and caravans — a caravan is just a “trade chance” upgrade roll on a visitor. Trade is bolted onto the visit system, so caravan frequency is chained to social-visit frequency rather than being its own economic loop.

Wildlife, spawning & husbandry

EntityService · entity/*

Creatures seed and spawn from lairs, run a per-tick brain (wander, hunt, hunger, flee), and leave corpses to butcher. That whole loop is live. Taming and husbandry are not.

◆ why is it like this?The schema commits to a feature the sim doesn't have: a TamedAnimal type, a tamedAnimals save slot, and a tameable flag on every creature — with zero code reading any of it. Speculative data-modelling ahead of behaviour, paid for in every save.

Stealth

core/stealth · ADR-032

A pawn in a mob's line of sight must also lose a perception-vs-stealth roll to be noticed; darkness and weather shorten sight. The maths is pure and lives in one small file; the roll is cached per mob for ~2 seconds.

◆ why is it like this?The detection cache is stored on the mob and cleared when it gives up a hunt. So stealth correctness is entangled with the hunt-FSM's give-up path — a bug in those transitions silently breaks re-stealthing.

The living world · seasons & weather

services/EnvironmentService (1,682 lines)

Light and tint cycle through the day; seasons bake a per-tile temperature that swings between noon and midnight; weather rolls and snow accumulates; fires project a heat field. All in one service.

◆ the clearest god-moduleOne file fuses five distinct domains: light, season, weather, a thermal/fire field, and gameplay crafting rules (fermentation & drying temperature windows). Those crafting windows sitting in a world-physics service is a plain misplacement — they leaked in because they happen to need temperature.

The map · grid & pathfinding

PathfinderService → spatial-core (Rust) · ADR-008

A* pathfinding and nearest-enemy queries are written in Rust, compiled to WASM, and reached only through an interface — callsites never touch the Rust. It even loads byte-identically under Node for headless replays.

◆ why is it like this?WASM loads asynchronously, but the sim tick is synchronous. Nothing guarantees the pathfinder is ready before the sim starts — so every callsite carries a defensive “is it ready? else fall back” branch. The impedance mismatch is smeared across all callers instead of fixed once at boot.

Research

services/ResearchService

A tiered tree; each node lists prerequisites that must all be complete, and finishing it unlocks buildings, items, abilities and a tool-tier bump.

◆ why is it like this?There's an empty systems/Research.ts — a one-line fossil of an abandoned systems→services migration that survives to mislead. And research applies unlocks by mutating state in place rather than the immutable-return style its neighbours use.

Audio

audio/AudioService · AudioController.svelte

A Howler wrapper plays; a reactive driver decides what — night track, fade the rain — from game state.

◆ why is it like this?All the audio orchestration logic lives in a Svelte component, so core sound behaviour is only alive while that component is mounted, and it reaches into environmentService from inside the UI layer. Sound decisions sit one floor too high.

The reason it's complex · how it grew

Almost nothing was born complex — it got specific under pressure

The single most useful thing to understand about this codebase: its complexity is sedimentary. Each system started as a generic placeholder and got specific only when a real need — a design spec, a performance ceiling, an audit finding — forced it. You can read that growth in the numbered spec generations.

Engine performance — each pass broke a ceiling the last one exposed

Pass I
Get the sim off the main thread
everything on one thread → a background worker, a slimmed snapshot, chunked terrain, render-on-demand. Tuned 38k tiles / a few hundred mobs to 200+ TPS.
Pass II · Jun 22
750² and 1,650 mobs exposed the next wall
TPS collapsed with mob count → two breakthroughs: an entity complexity bubble (throttle mob thinking by distance, not drawing) and incremental-only terrain (repaint only changed cells, ADR-026). → 120 FPS / 60 TPS on a 500×500 map.
Pass III · Jul 16
A winter save found two more
per-mob light recomputed every tick; a snow scan that froze at 4× → a baked point-light field (an O(1) field read replacing a loop over every light source per mob), and chunked snow accumulation.
THE METHOD, NOT JUST THE FIX

Every pass was driven by measurement, never guessing — CPU traces and A/B toggles that disproved suspects (“it's stealth,” “it's the snow chunk”) before finding the real cost. That discipline is itself part of the architecture: the headless profiler and the trace methodology are permanent tools, not one-offs.

And the smaller lineages

Lineages I → II

Ancestral-blood mutation trees, then transforms and feeding — folded into the trait system so traits became the single source of truth.

Combat → Creature-Combat

A human-vs-human wound model, then a layered-armour overhaul so creatures fight with weird bodies, status procs and no strength bar.

DF-migration → today

An early proposal to become Dwarf-Fortress-like is the DNA of today's deep anatomy and tile model.

The Rust / Dioxus port

A full-rewrite evaluation — parked. The WASM spatial core is the pragmatic compromise the perf passes chose instead.

The open questions the map surfaced

Drawn from AUDIT.md, codegraph, and the code sweeps behind every ◆ above.

◆ structureGrowth is the “once a day” grab-bag — offers + aging + lineage, mutated in place. Split the daily hook from stat growth?
◆ structurePawnStateMachine & Combat never finished decomposing — medical sim + lineage credit still live inside them.
◆ correctnessThe death rule runs in 3 places; passivity in 2. Single-source them?
◆ namingA new furniture comfort meter collides with the existing temperature “comfort band.”
◆ contentCrystals collapse into one gem_dust; voidshard 0 uses, arcane_resin 0 consumers — identity washed out.
◆ realismSteel recipes are inverted — finery decarburises, blast makes pig iron. The chain rework is already tracked.
