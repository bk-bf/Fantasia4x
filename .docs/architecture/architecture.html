Click a tab → it flips one value → the app swaps in that screen. Nothing shows at all until the game has finished loading.

In the code

The single value is uiState.currentScreen. A plain list of “if it's this, show that” in +page.svelte picks the screen. Clicking a tab calls setScreen(). The loading gate is appPhase going from 'menu' to 'game'.

The screens themselves

Pawns · Work · Building · Crafting · Exploration · Culture · Kingdoms · Research · Entities · Debug

The map itself is drawn by one big canvas component, separate from the menus.

The screens are just viewers. They never change the world directly — they only ask for it to change (next slides).

Whenever the picture updates, it's quietly written to a small database in the browser saveManager → IndexedDB. Autosave rides along.

2½The wall in the middle

It only sends what actually changed

Shipping the entire world 15 times a second was too heavy — it was a third of the game's whole workload. So each update carries only the differences.

the whole world
mostly unchanged
→
only the changed bits
this pawn moved · that tree fell · this pile shrank
→
re-stitched
on the other side

A pawn that didn't move sends nothing. A carcass rotting on the ground sends nothing until it changes. That “send nothing when nothing happened” rule is what keeps the frame rate up.

In the code: a per-thing “what changed since last time” diff (EntitySync) reassembled by simWorkerClient.

2½The one doorway to change things

Every possible action — chop, equip, draft, research, haul — is a named entry in one list. The request is just data (“chop tree #12”), so it can safely cross the wall and be replayed.

WHY IT'S BUILT THIS WAY

One doorway, three users: the buttons you click, the automated test scripts, and the dev tools all send the exact same requests. Same requests in the same order always produce the exact same game — which is what makes bug-hunting and replays possible.

In the code: the list is COMMANDS in commands.ts (88 entries) · e.g. chop = designate, equip = equipPawnItem, draft = toggleDraft.

4Two folders that look similar

Does it drive the clock, or does it answer a question?

Drives the clock — systems/

The few things that push time forward or reach across everything at once:

  • the turn-taker gameEngine
  • each pawn's moment-to-moment state PawnStateMachine
  • the blow-by-blow of fights Combat
  • totalling up every bonus ModifierSystem

Answers questions — services/

Everything else. Give it the world and a question, it gives you an answer or an updated world.

“How much wood is here?” “Can this be crafted?” “Where's the nearest deer?” Each stays inside its own topic, and the drivers above call on them.

Rough rule of thumb — not a hard wall. A couple of files sit in the “wrong” folder for historical reasons.

5The vault + the rulebook

Every change to the world goes through one gate

add wood
hurt this pawn
place a building
finish a job
→
the gatekeeper
GameStateManager

Nothing writes to the world on its own. It all funnels through this one gate — so there's always a single, known place where reality changes.

The rulebook is data, not code

Creatures, items, recipes, traits, buildings — all live in big data files database/*.jsonc. Code just loads them. Adding content usually means editing data, not writing logic.

The shapes of everything

What a pawn, a building, an item is — the definitions live in core/types/.

Heads-up: the old notes point at core/Items.ts and Buildings.ts — those are gone. It's all database/*.jsonc now.

Four rules that hold everywhere

When there's one right way, there's only one way

“How good is this pawn at this job?”

Answered in exactly one place, from their body and traits. There is no second opinion. pawnStatService.getWorkModifiers

“What bonuses apply right now?”

One tallier adds up gear, buildings and traits — and always shows its receipts. ModifierSystem

“How do things move across the map?”

The heavy path-finding math is written in Rust for speed, reached through one tidy door. PathfinderService → WASM

“Can it run with no window open?”

Yes — the same game, driven by scripts instead of clicks, for tests and replays. headless sim

Each of these has burned in a “don't build a second one of these” rule. It's why a new feature is usually new data, not a new code path.

Finding your way around

Two threads to pull, five knots to know

Follow a click

button → a named request → across the wall → looked up in the list → the gatekeeper changes the world → a fresh picture comes back → the screen redraws.

Follow a moment of time

the turn-taker → its 19 steps → each specialist does its bit → the world moves one tick forward.

The five knots — learn these and the rest untangles

the one gate that changes the world GameStateManager · the one doorway of requests commands.ts · the one answer to “how good at this job” getWorkModifiers · the one bonus-tallier ModifierSystem · the one door to map-movement PathfinderService.

One gate changes the world. One doorway carries every request. One answer to each question.
