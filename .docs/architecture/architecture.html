2½The worker boundary  ADR-021

The sim runs off the main thread

  • The worker owns the canonical GameState and runs the whole tick loop off-thread sim.worker.ts:2–8, :536. It constructs the GameStateManager and drives gameEngine.processGameTurn() in a batch() loop.
  • The main-thread store is a read-only projection gameState.ts:1207 — setSilent(v) every tick so get(gameState) is current for the renderer; notify() + save fire only on flush :1677.
  • The engine never imports the store — it publishes through an injected outputSink sim.worker.ts:600.

2½The worker boundary

What actually ships each flush

state — sectional diff

Only top-level fields whose ref changed. pawns/mobs/worldMap/droppedItems excluded (SECTIONAL_SKIP); 3 synthetic revs _terrainRev/_snowRev/_designationRev forced.

pawns / mobs — EntitySync

Per-entity: slim hot projection + any cold field whose ref changed (PAWN_COLD/MOB_COLD). Mob needs ships every flush; removed[] reaps despawns.

drops — per-id ref diff

Only stacks whose object ref changed ship; an unchanging pile ships nothing — the carcass-FPS fix.

worldMap / worldMapDelta

Full tile array only on ref change (worldgen/load); otherwise slim in-place tile deltas (TILE_RENDER_FIELDS), snow tagged k:1, budgeted.

Consumer simWorkerClient merges the diff onto lastState, rebuilds pawns/mobs/drops from per-id mirrors via applyEntitySync, reattaches the cached worldMap, then onState(state, flush). commit frames apply even while paused.

Producer: sim.worker.ts:362 (publish) · protocol: simProtocol.ts (EntitySync :18)

2½Commands — the write path  ADR-033

Player & dev actions are serializable commands

  • Registry COMMANDS: Record<string,Cmd> — 88 entries commands.ts:189. Each is a pure (state, payload) ⇒ GameState.
  • A command is {type, payload?, save?} — no closures, so it survives structured-clone across the worker boundary simProtocol.ts:22.
  • Dispatch: UI → gameState.command(cmd) → dispatchCommand :1246. Worker mode posts {kind:'command'}; else applies in-thread.
  • Worker: applyCommand(s ⇒ applySimCommand(s, cmd)) sim.worker.ts:621 → looks up COMMANDS[cmd.type].

Example commands

addItem · toggleDraft · attackTargetWith · equipPawnItem · designate · startResearch · haulTileToStockpile

dev: devSpawnPawns · devSpawnAllItems · devSpawnItem

ONE REGISTRY, THREE FRONTS

The same commands.ts is steered by the UI, the dev-only /api/sim HTTP driver, and the invariant test suite. Same seed + scenario + command script ⇒ byte-identical state (no Date.now() in ids).

3Turn coordinator  GameEngineImpl.ts:263

One tick = processGameTurn()

  • gameEngine is the orchestrator, not the logic — it sequences service calls and owns the sinks.
  • Each tick: shallow-copy state + bump counter :278 → run phases through a timed() perf wrapper → GameStateManager.updateState() :387 → wall-clock-gated push ~15 Hz :393.
  • 19 phases, fixed order — later phases read what earlier ones wrote. Some are throttled (run every N ticks).
needs→ item upkeep→ research→ jobs→ buildings→ pawns→ growth→ entities→ combat→ reap→ events

The order is load-bearing. Do not reorder.

Throttles: jobs /6 · decay & drying /60 · deterioration /600 · dialog /90 ticks.

BARREL IS PARTIAL

services/index.ts wires only 6 (itemService, buildingService, workService, researchService, modifierSystem, gameEngine). The other ~20 are imported directly — no runtime DI, just module-level export const. The ServiceIntegration.ts the barrel references no longer exists.

Each singleton implements an interface (PawnService, ItemService, PathfinderService…) and exports one instance.

4systems/ vs services/

Two folders, one rule of thumb

systems/ — drives the loop / aggregates across every domain

  • gameEngine — the tick pipeline + sinks
  • pawnStateMachineService — advances every pawn's condition FSM each tick :1476
  • combatService — real-time tickCombat(dtMs) (2608 lines)
  • modifierSystem — fans out over buildings/items/traits/research
  • gameCoordinator — the odd one: a tiny UI facade

services/ — stateless domain helpers keyed on GameState

Almost all are (…args, gameState) ⇒ GameState or pure lookups scoped to a single domain (items, jobs, pawns, kingdoms, resources, lighting).

They are consumed by the systems layer — the engine drives pawnService/jobService/entityService; gameCoordinator delegates to itemService/buildingService.

NAMING QUIRKS IN THE ACTUAL CODE

services/MovementSystem.ts is named "System" but lives under services/ and exports only functions · PathfinderService.ts is an interface + alias, not a class · systems/Diplomacy.ts · Events.ts · Research.ts are 1-line stubs (real research = services/ResearchService.ts).

Cross-cutting

Four systems that touch every layer

ModifierSystem

Aggregates building + item + equipment + trait + research bonuses. Every result carries sources[]. Never hand-sum bonuses when a method exists.

Single work model ADR-015

Work speed/yield/quality lives ONLY in pawnStatService.getWorkModifiers (stats.jsonc × body capacities × trait effects). The old calculateWorkEfficiency path was deleted — don't reintroduce a second one.

Spatial = WASM behind an interface ADR-008

A* pathfinding, nearest-entity, fog-of-war → Rust→WASM via wasmPathfinderService. Callsites use only the PathfinderService interface, never import spatial-core/.

Headless sim ADR-033

In-thread driver over the same engine + the 88-command registry: /api/sim HTTP, the DEBUG godmode panel, and the invariant tests. Byte-identical replay.

These are the invariants that keep the layers honest — the reason a feature rarely needs a new code path, just new data.

  • gameStateManager — only mutation surface
